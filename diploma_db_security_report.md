# Разделы дипломного проекта мобильного приложения «SpotFynder»

В данном файле представлены готовые подразделы для дипломной работы, детально описывающие механизмы автоматизации на уровне базы данных MySQL (триггеры) и комплексный подход к защите информации в разработанной архитектуре (FastAPI + SQLAlchemy + MySQL + Flutter).

---

## Раздел 1. Триггеры базы данных (MySQL)

Для обеспечения целостности данных, автоматизации рутинных операций и реализации бизнес-логики непосредственно на уровне СУБД MySQL, в базу данных мобильного приложения **«SpotFynder»** были интегрированы специализированные SQL-триггеры. 

Использование триггеров позволяет гарантировать выполнение критически важных правил независимо от клиентского приложения или серверного кода, разгружая backend-сервер и повышая надежность системы.

### 2. Триггер для валидации оценки в комментариях
В таблице `comment_places` предусмотрено поле `estimation` (оценка арт-объекта от 1 до 5). Для предотвращения записи некорректных значений (например, из-за сбоя в API или обхода ограничений интерфейса) на уровне базы данных реализован триггер жесткой валидации данных перед вставкой (`BEFORE INSERT`).

#### SQL-код триггера:
```sql
DELIMITER $$

CREATE TRIGGER before_comment_places_insert
BEFORE INSERT ON comment_places
FOR EACH ROW
BEGIN
    -- Проверка корректности диапазона оценки (от 1 до 5)
    IF NEW.estimation IS NOT NULL AND (NEW.estimation < 1 OR NEW.estimation > 5) THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Ошибка валидации: Оценка (estimation) должна быть целым числом в диапазоне от 1 до 5.';
    END IF;
END$$

DELIMITER ;
```

#### Логика работы:
* Перед вставкой новой записи в таблицу `comment_places` триггер анализирует значение поля `NEW.estimation`.
* Если переданное значение выходит за рамки от 1 до 5, триггер вызывает прерывание транзакции с помощью команды `SIGNAL SQLSTATE '45000'` (пользовательское исключение в MySQL).
* Это отменяет операцию вставки и возвращает клиенту (FastAPI) понятное сообщение об ошибке, защищая БД от замусоривания.

### 3. Триггер для предотвращения удаления активных пользователей (защита связанных данных)
В системе «SpotFynder» пользователи (`users`) могут создавать уникальные точки стрит-арта и достопримечательности (`places`). В случае удаления аккаунта автора, удаление связанных с ним мест может привести к потере ценного контента. 
Для защиты от случайного удаления пользователей с активными арт-объектами разработан триггер, блокирующий операцию удаления.

#### SQL-код триггера:
```sql
DELIMITER $$

CREATE TRIGGER before_users_delete
BEFORE DELETE ON users
FOR EACH ROW
BEGIN
    DECLARE active_places_count INT;
    
    -- Подсчет количества мест, созданных удаляемым пользователем
    SELECT COUNT(*) INTO active_places_count 
    FROM places 
    WHERE id_user = OLD.id;
    
    -- Блокировка удаления при наличии созданных мест
    IF active_places_count > 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Удаление невозможно: пользователь является автором зарегистрированных арт-объектов (мест).';
    END IF;
END$$

DELIMITER ;
```

#### Логика работы:
* Триггер срабатывает **перед (`BEFORE`)** удалением записи из таблицы `users`.
* Ссылка `OLD` указывает на удаляемую запись пользователя.
* С помощью SQL-запроса проверяется наличие записей в таблице `places`, где `id_user` равен `OLD.id`.
* Если найдено хотя бы одно место (`active_places_count > 0`), транзакция откатывается (`ROLLBACK`), возвращая ошибку ограничения целостности на уровне бизнес-логики.

## Раздел 2. Защита информации в базе данных и архитектуре приложения

Безопасность персональных и системных данных в проекте **«SpotFynder»** обеспечивается комплексным подходом на всех уровнях архитектуры: СУБД MySQL, backend-платформа FastAPI и мобильный клиент Flutter.

### 1. Хеширование паролей с использованием алгоритма bcrypt
Хеширование паролей пользователей в открытом виде недопустимо. В системе «SpotFynder» пароли подвергаются одностороннему криптографическому хешированию на стороне сервера перед записью в поле `hashed_password` таблицы `users`.

* **Механизм**: Для хеширования применяется стандартная библиотека Python `passlib` с алгоритмом `bcrypt`.
* **Принцип работы**: 
  1. При регистрации или смене пароля сервер генерирует уникальную случайную соль (salt) для каждого пароля.
  2. Пароль объединяется с солью и многократно хешируется.
  3. Сгенерированный хеш-строка сохраняется в базу данных.
  4. При аутентификации введенный пароль проверяется функцией `bcrypt.verify()`, сравнивающей введенное значение с сохраненным хешем.
* **Результат**: Защита от атак по радужным таблицам (rainbow tables) и полного перебора (brute force) даже в случае полной компрометации (утечки) базы данных.

### 2. Безопасность и авторизация по протоколу JWT (JSON Web Tokens)
Взаимодействие мобильного клиента Flutter с FastAPI-сервером построено на принципах REST API без сохранения состояния сессии (stateless).

* **Процесс аутентификации**:
  1. Пользователь отправляет свои учетные данные (email, пароль) на эндпоинт `/auth/login`.
  2. После успешной проверки FastAPI генерирует access-токен в формате **JWT**, содержащий `sub` (идентификатор пользователя) и время жизни токена (`exp`). Токен подписывается секретным ключом сервера с алгоритмом `HS256`.
  3. Flutter-клиент сохраняет JWT в защищенное локальное хранилище (`flutter_secure_storage`).
* **Авторизация запросов**:
  При каждом последующем запросе клиент прикрепляет JWT-токен в HTTP-заголовок `Authorization: Bearer <token>`. Сервер валидирует подпись токена. Если подпись верна и токен не истек, запрос успешно обрабатывается.

### 3. Разграничение прав доступа на уровне сервера (RBAC)
Доступ к различным ресурсам и операциям разграничивается с помощью ролевой модели (Role-Based Access Control). В СУБД существует таблица `roles` с ролями: **гость, авторизованный, модератор, администратор**.

* На стороне FastAPI безопасность реализуется через механизм **Dependency Injection (внедрение зависимостей)**.
* Создаются специальные функции-депенденси (например, `get_current_active_user`, `require_moderator_role`), которые извлекают роль текущего пользователя из базы данных и проверяют соответствие требованиям эндпоинта.
* **Пример**: Обычный авторизованный пользователь может создавать новые места (`places`) и оставлять отзывы, но только пользователь с ролью `moderator` или `admin` имеет доступ к изменению статусов мест (`id_status`) или обработке жалоб (`report_places`). Гости имеют права только на чтение общедоступной информации.

### 4. Предотвращение SQL-инъекций (SQL Injection Prevention)
SQL-инъекция — одна из наиболее опасных угроз для реляционных баз данных. В архитектуре «SpotFynder» эта уязвимость полностью нейтрализована благодаря использованию объектно-реляционного отображения (**ORM SQLAlchemy**).

* **Принцип защиты**: SQLAlchemy взаимодействует с базой данных MySQL исключительно через **параметризованные запросы** (prepared statements).
* При выполнении запросов пользовательский ввод передается отдельно от структуры самого SQL-запроса в качестве параметров. Драйвер базы данных автоматически экранирует спецсимволы и кавычки.
* **Результат**: Любой внедренный вредоносный код (например, `' OR '1'='1`) воспринимается СУБД как обычная строка (литерал), а не исполняемая инструкция, что делает проведение SQL-инъекций невозможным.
